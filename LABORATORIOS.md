# Laboratorios prácticos

## Laboratorio 1 - Primer aporte al repositorio

1. Clonar el repositorio
2. Crear una nueva rama en tu máquina local.
   **Nombre sugerido de la rama:** feature/tu-nombre
3. Crear una carpeta dentro de `participantes` con tu nombre
4. Crear un archivo `info.md`
5. Agregar la siguiente información:
   - Nombre
   - Rol o cargo
   - Experiencia utilizando Git o GitHub  
6. Guardar los cambios.
7. Enviar tu rama al repositorio remoto.

# Laboratorio 2 - Trabajo con múltiples ramas y resolución de conflictos

1. Cambiar nuevamente a la rama `main` en tu repositorio local.
2. Crear una nueva rama desde `main`.
   **Nombre sugerido de la rama:** feature/tu-nombre-v2
3. Crear nuevamente una carpeta dentro de `participantes` con tu nombre y el archivo `info.md`.
4. Agregar la siguiente información:
   - Nombre
   - Lenguajes de programación que conoces
   - Herramientas tecnológicas que utilizas en tu trabajo 
5. Guardar los cambios en esta nueva rama.
6. Cambiar nuevamente a la rama creada en el laboratorio 1.
7. Intentar integrar los cambios de la nueva rama en esta rama.
8. Analizar el conflicto generado en el archivo `info.md`.
9. Resolver el conflicto manualmente combinando la información de ambas ramas para que el archivo final incluya:
    - Nombre
    - Rol o cargo
    - Experiencia utilizando Git o GitHub
    - Lenguajes de programación
    - Herramientas tecnológicas que utilizas en tu trabajo
10. Enviar tu rama al repositorio remoto.

## Laboratorio 3 - Pull Request y revisión de código

1. Crear un Pull Request desde tu rama hacia la rama `main`.
2. Asignar como revisores a todos los integrantes del grupo.
3. Revisar un solo Pull Request asignado por otro compañero.
4. Realizar al menos dos tipos de comentarios:
   - Un comentario general en la conversación del Pull Request (por ejemplo, felicitando el trabajo, dando una opinión general o sugiriendo mejoras).
   - Un comentario específico en alguna parte del archivo revisado.
5. Responder a los comentarios recibidos en tu Pull Request si es necesario.
6. Aprobar el Pull Request donde fuiste asignado como revisor cuando consideres que los cambios están correctos.
7. Realizar el merge del Pull Request aprobado.

## Laboratorio 4 - Trabajo con Issues y GitHub Projects

1. Revisar la lista de Issues disponibles en el repositorio.
2. Seleccionar un Issue que no esté asignado.
3. Asignarte el Issue seleccionado.
4. Agregar el Issue al Project del repositorio en la columna correspondiente.
5. Mover el Issue a la columna **In progress** cuando comiences a trabajar en él.
6. Crear una nueva rama relacionada con el Issue.
   **Nombre sugerido:** feature/issue-numero-descripcion
7. Realizar los cambios solicitados en el Issue.
8. Crear un Pull Request relacionado con el Issue.
9. En la descripción del Pull Request indicar que el Issue será cerrado cuando se haga merge.
10. Una vez aprobado el Pull Request, visualizar que el Issue se encuentra en la columna "Done" del proyecto.

## Laboratorio 5 - Resolución de conflictos

1. Crear una nueva rama a partir de la rama main.
   **Nombre sugerido:** feature/update-by-tu-nombre
2. Modificar la última línea del archivo `README.md` colocando tu nombre.
3. Guardar cambios y subir al repositorio remoto.
4. Crear un Pull Request desde la nueva rama hacia `main` y agregan únicamente como revisor al usuario K23Tatiana.
5. Cuando aparezca un conflicto de merge, analizar el conflicto generado.
6. Resolver manualmente el conflicto para conservar los cambios de todos los integrantes.
7. Actualizar el Pull Request con la solución.
8. Completar el proceso de revisión y merge.
